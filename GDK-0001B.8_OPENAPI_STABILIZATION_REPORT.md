## Problem Analysis

### Root Cause: Enum Type Inconsistency

The error `prop.enum.map is not a function` occurred because enum values reached the renderer in multiple inconsistent forms:

1. **Inline Array Syntax in YAML**: 
   ```yaml
   values: ['EQUIPMENT', 'VEHICLE', 'BUILDING']
   ```
   SimpleYAMLParser outputs this as a **string**: `"['EQUIPMENT', 'VEHICLE', 'BUILDING']"`

2. **Proper Array in YAML**:
   ```yaml
   status:
     - PROSPECT
     - ACTIVE
     - INACTIVE
   ```
   This outputs as an actual JavaScript array: `['PROSPECT', 'ACTIVE', 'INACTIVE']`

3. **Undefined/Null Values**: Fields with missing `values` property passed `undefined` or `null`

4. **Non-Array Types**: Potential for other types to slip through normalization

### Failure Point: OpenAPIRenderer

In OpenAPIRenderer, lines like:
```javascript
if (prop.enum) lines.push(`          enum: [${prop.enum.map(v => `"${v}"`).join(', ')}]`);
```

This code assumes `prop.enum` is always an array. When it's a string (from inline array syntax), `.map()` fails with "not a function".

### Root Cause Chain

1. **SimpleYAMLParser**: Outputs inline array syntax as strings
2. **FieldExpander**: Didn't normalize enum values (assumed they'd be arrays)
3. **OpenAPIRenderer**: Called `.map()` on unvalidated enum property without defensive checks

## Implementation Approach

### Dual-Level Normalization Strategy

Similar to PolicyRenderer fix, implemented normalization at **both** levels:

#### 1. **FieldExpander Level** (Earliest Canonical Boundary)

Added comprehensive enum normalization in [tools/genesis/compiler/metadata-engine/FieldExpander.mjs](tools/genesis/compiler/metadata-engine/FieldExpander.mjs#L40-L62):

```javascript
if (fieldDef.type === 'enum') {
  // Normalize enum values: handle string (inline array syntax), array, or undefined
  let values = fieldDef.values;
  if (typeof values === 'string') {
    // Parse inline array syntax: "['ENUM1', 'ENUM2']" or "[ENUM1, ENUM2]"
    if (values.startsWith('[') && values.endsWith(']')) {
      values = values
        .slice(1, -1) // Remove brackets
        .split(',')
        .map(v => v.trim().replace(/^['"]|['"]$/g, '')) // Remove quotes and trim
        .filter(v => v); // Remove empty strings
    } else {
      values = [values];
    }
  } else if (!Array.isArray(values)) {
    values = [];
  }
  
  // Deduplicate and sort for determinism
  expanded.values = Object.freeze(
    Array.from(new Set(values.map(v => String(v)))).sort()
  );
  expanded.default = fieldDef.default;
}
```

**Features:**
- Detects inline array syntax via `startsWith('[')` and `endsWith(']')`
- Removes brackets and quote characters
- Converts single strings to arrays
- Deduplicates via Set
- Sorts alphabetically for determinism
- Freezes for immutability

#### 2. **OpenAPIRenderer Level** (Defensive Normalization)

Added defensive normalization utilities in [tools/genesis/compiler/renderers/OpenAPIRenderer.mjs](tools/genesis/compiler/renderers/OpenAPIRenderer.mjs#L19-L56):

```javascript
function normalizeEnumValues(input) {
  if (!input) return [];
  
  if (typeof input === 'string') {
    // Handle inline array syntax
    if (input.startsWith('[') && input.endsWith(']')) {
      const parsed = input
        .slice(1, -1)
        .split(',')
        .map(v => v.trim().replace(/^['"]|['"]$/g, ''))
        .filter(v => v);
      return Array.from(new Set(parsed)).sort();
    }
    // Single string value
    return [input];
  }
  
  if (Array.isArray(input)) {
    const values = input.map(v => String(v)).filter(v => v);
    return Array.from(new Set(values)).sort();
  }
  
  return [];
}
```

**Benefits:**
- Never assumes type
- Tolerates any input form
- Always returns clean array
- Never crashes on malformed data

#### 3. **Safe Property Rendering**

Added `renderFieldProperty()` function that uses safe checks before accessing any properties:

```javascript
function renderFieldProperty(field) {
  const lines = [];
  const prop = fieldToOpenAPIProperty(field);
  
  if (!prop) return lines;
  
  // Type
  if (prop.type) {
    lines.push(`          type: ${prop.type}`);
  }
  
  // ... other properties ...
  
  // Enum values (always normalized and sorted)
  if (Array.isArray(prop.enum) && prop.enum.length > 0) {
    const enumStr = prop.enum.map(v => `"${v}"`).join(', ');
    lines.push(`          enum: [${enumStr}]`);
  }
  
  return lines;
}
```

## Key Learnings

### 1. **Type Consistency is Critical**
Inconsistent types propagating through the pipeline (string vs array enum values) created runtime crashes. Normalization at the earliest boundary prevents downstream issues.

**Lesson**: Normalize data types at the metadata source; don't assume consistent types downstream.

### 2. **Inline Array Syntax is Common in YAML**
Many developers use inline array syntax like `[VAL1, VAL2]` instead of YAML list syntax. YAML parsers handle this differently, sometimes returning strings.

**Lesson**: Anticipate and handle inline array syntax parsing explicitly.

### 3. **Determinism Requires Sorting Everything**
Even with normalized types, iteration order matters. JavaScript object property order may be insertion-order in modern JS, but explicit sorting is safer and more portable.

**Lesson**: Always sort when determinism matters; don't rely on iteration order.

### 4. **Dual-Level Normalization Provides Defense in Depth**
Normalizing at both expander and renderer levels creates resilience. If expander normalization is incomplete, renderer doesn't crash.

**Lesson**: Implement normalization twice - once at source, once at consumption point.
