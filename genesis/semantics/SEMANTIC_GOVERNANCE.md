# Semantic Governance

Status: Approved
Classification: Genesis Standard

## Purpose

This document defines how semantic standards are proposed, reviewed, approved, versioned, and retired.
It governs semantic evolution through evidence and verification, not preference.

## Proposal Lifecycle

1. Draft
2. Experimental
3. Candidate
4. Approved
5. Canonical
6. Deprecated
7. Archived

## Review Process

Review must verify:
- derivation integrity from seed concepts
- evidence quality and lineage completeness
- constitutional compliance
- compiler compatibility impact
- backward compatibility implications

Review outputs:
- approve
- conditional approve with amendments
- return for revision
- reject

## Approval Process

A proposal can be approved only when:
- required artifacts are complete
- evidence and verification are satisfactory
- conflicts are resolved
- reviewers record explicit decision and rationale

Canonical promotion requires additional validation evidence beyond initial approval.

## Semantic Versioning

Semantic versions follow major.minor.patch:
- major: breaking semantic meaning or compatibility change
- minor: backward-compatible semantic extension or clarification
- patch: non-semantic editorial correction

## Deprecation

Deprecation requires:
- rationale and impact statement
- migration guidance
- sunset timeline
- compatibility strategy during transition

Deprecated semantics remain documented until archival criteria are met.

## Backward Compatibility

Compatibility expectations:
- canonical IDs remain stable
- previously valid derived mappings continue to resolve unless explicitly version-gated
- compatibility breaks require major version increment and migration plan

## Compiler Compatibility

Semantic standards must define compiler implications at a specification level.
No semantic change can claim compatibility without validation against compiler constraints.
Implementation changes are tracked in compiler specifications, not in this standards document.

## Constitutional Compliance

All semantic standards must align with Genesis First Principles and Genesis Scientific Method.
If conflict is detected, constitutional doctrine and governance method are authoritative.

## Extension Approval

Extension approvals require:
- explicit parent mapping to core semantics
- proof that extension does not violate core invariants
- evidence package and verification report
- compatibility statement for existing canonical concepts

Extensions failing these criteria remain non-canonical.

## Rejection Conditions

A proposal is rejected when:
- derivation is missing or invalid
- evidence is insufficient or non-traceable
- compatibility impact is unbounded
- constitutional conflict is unresolved
