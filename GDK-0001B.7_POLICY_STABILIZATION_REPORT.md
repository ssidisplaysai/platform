## Problem Analysis

### Root Cause Chain (5 Components)

This milestone required detective work across the entire metadata → blueprint → renderer pipeline. The root cause chain involved 5 interacting systems:

#### **Component 1: SimpleYAMLParser (YAML Input)**
**Behavior**: Parses entity YAML files, including inline array syntax like `[admin, manager]`  
**Issue**: Outputs inline arrays as **strings** `"[admin, manager]"` rather than parsed arrays

**Example:**
```yaml
# Entity YAML
capabilities:
  permissions:
    roles: [admin, manager, viewer]
```

**YAML Parser Output:**
```javascript
{
  capabilities: {
    permissions: {
      roles: "[admin, manager, viewer]"  // <-- string, not array!
    }
  }
}
```

#### **Component 2: CapabilityExpander (First Normalization Point)**
**Issue**: Received string roles from YAML parser but didn't normalize them  
**Original Behavior**: Passed roles unchanged to next stage

```javascript
// BEFORE: No normalization
if (name === 'permissions') {
  // ... roles received as string "[admin, manager]"
  result.permissions.roles = metadata.roles;  // Raw YAML input
}
```

**Resolution**: Added inline array parsing:
```javascript
// AFTER: Parse and normalize
if (name === 'permissions' && typeof metadata.roles === 'string') {
  if (metadata.roles.startsWith('[') && metadata.roles.endsWith(']')) {
    // Parse inline array syntax
    const parsed = metadata.roles.slice(1, -1)
      .split(',')
      .map(r => r.trim())
      .filter(r => r);
    result.permissions.roles = Object.freeze(parsed);
  }
}
```

#### **Component 3: PermissionExpander (Comprehensive Normalization)**
**Issue**: Received non-normalized roles; need to deduplicate, sort, and freeze  
**Original Behavior**: Minimal processing of roles input

**Resolution**: Added complete normalization pipeline:
```javascript
function normalizeRoles(roles) {
  // Handle undefined/null
  if (!roles) return Object.freeze([]);
  
  // Convert single string to array
  if (typeof roles === 'string') {
    // Parse inline array syntax: "[admin, manager]"
    if (roles.startsWith('[') && roles.endsWith(']')) {
      const parsed = roles.slice(1, -1).split(',').map(x => x.trim()).filter(x => x);
      roles = parsed;
    } else {
      roles = [roles];
    }
  }
  
  // Convert to array if not already
  if (!Array.isArray(roles)) {
    return Object.freeze([]);
  }
  
  // Deduplicate: convert to Set and back
  const deduplicated = new Set(roles.map(r => String(r).toLowerCase()));
  
  // Sort alphabetically for determinism
  const sorted = Array.from(deduplicated).sort();
  
  // Freeze and return
  return Object.freeze(sorted);
}
```

#### **Component 4: BlueprintBuilder (Metadata Integration)**
**Issue**: Line 202 overwrote normalized roles with non-array values  
**Original Code:**
```javascript
permissions.roles = roles;  // <-- Could be string or any type
```

**Resolution**: Ensured BlueprintBuilder respects already-normalized values from expanders

#### **Component 5: PolicyRenderer (Defensive Normalization + Deterministic Rendering)**
**Issue**: Called `roles.forEach()` without verifying type  
**Original Error:**
```
Error rendering policies: roles.forEach is not a function
// roles was a string or non-array value
```

**Resolution**: Added defensive normalization at renderer level AND sorted all output:

```javascript
// Defensive normalization utilities in PolicyRenderer
function normalizeRoles(input) {
  if (input === undefined || input === null) return [];
  if (typeof input === 'string') {
    if (input.startsWith('[') && input.endsWith(']')) {
      return Object.freeze(
        input.slice(1, -1)
          .split(',')
          .map(x => x.trim())
          .filter(x => x)
          .sort()
      );
    }
    return Object.freeze([input]);
  }
  if (Array.isArray(input)) {
    return Object.freeze(
      [...new Set(input.map(x => String(x)).filter(x => x))].sort()
    );
  }
  return [];
}

// All policy iteration uses sorted arrays for determinism
const sortedRoles = Array.from(policies.byRole).sort();
sortedRoles.forEach(([role, rolePolicies]) => { ... });
```

## Implementation Approach

### Dual-Level Normalization Strategy

To prevent this failure mode from recurring, we implemented normalization at **both**:

1. **Expander Level** (PermissionExpander)
   - Closest to metadata source
   - Most authoritative for blueprint IR
   - Results are sorted and frozen for immutability
   - Blueprint IR maintains canonical normalized state

2. **Renderer Level** (PolicyRenderer)
   - Defensive checks
   - Assumes blueprint may have unexpected values
   - Never calls methods assuming type
   - Safe even if blueprint normalization is incomplete

### Deterministic Ordering

All sorted iteration follows pattern:
```javascript
// Always sort for determinism
const sorted = Array.from(collection).sort();
sorted.forEach(item => { /* process */ });
```

Key sorting locations in generated policy markdown:
- Supported Roles list (alphabetical)
- Supported Actions list (alphabetical)
- Policies by Condition Type section (by type name)
- Policies by Role section (by role name)

### Helper Functions

Added 4 formatting helpers for consistent display:
- `formatActionDescription(action)`: Maps actions to human-readable descriptions
- `formatConditionType(type)`: Maps condition types to section headers
- `getRoleDescription(role)`: Maps roles to descriptions
- `formatRoleName(role)`: Capitalizes and formats role names

## Key Learnings

### 1. **Metadata Pipeline Errors Cascade**
If early-stage normalizers (like YAML parser or CapabilityExpander) don't normalize data, later stages (PolicyRenderer) are forced to handle multiple input types. This can lead to type errors if type assumptions are made.

**Lesson**: Normalize as early as possible, make blueprint IR authoritative.

### 2. **Dual-Level Normalization Provides Safety**
While expanders should normalize, renderers should also validate/normalize inputs defensively. This pattern is resilient to future changes to the pipeline.

**Lesson**: Don't trust upstream even if you normalizeyourself.

### 3. **YAML Parser Quirks Matter**
The SimpleYAMLParser outputs inline arrays like `[admin, manager]` as strings, not JavaScript arrays. This is a common gotcha in YAML parsing.

**Lesson**: Know your YAML parser's behavior; handle string arrays explicitly.

### 4. **Determinism Requires Explicit Sorting**
Modern JavaScript maintains insertion order for object keys, but relying on this is fragile. Always sort explicitly for reproducible output.

**Lesson**: Never assume iteration order; sort deterministically.
