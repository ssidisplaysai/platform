## 1. Executive Summary

**M1.4 Semantic Resolution (BGC-PASS-005)** implements the first governed semantic transformation in the Business Genome Compiler pipeline. The pass deterministically resolves correlated evidence clusters into evidence-backed semantic candidates, proven by 81+ compiler-proof assertions across 8 invariant categories.

**Key Achievement:** Transform `CorrelatedEvidenceCollection` → `SemanticCandidateCollection` with **deterministic, verifiable, conflict-preserving** semantics. No consolidation, no relationships, no genome publication — M1.4 is a **pure semantic classification layer**.

## 4. Semantic Candidate Model

### 4.1 Data Structure (9 Interfaces, 15 Fields)

**SemanticCandidate** (primary output type):
```typescript
interface SemanticCandidate {
  readonly id: string;  // Deterministic hash(semanticClass + designation + evidence)
  readonly semanticClass: string;  // BGS-0001 semantic class (13 supported in M1.4)
  readonly designation: string;  // Entity name from evidence metadata
  readonly assertions: SemanticAssertion[];  // Evidence-backed claims
  readonly evidenceClusterIds: string[];  // Lineage: which clusters contributed
  readonly evidenceGroupIds: string[];  // Lineage: groups within clusters
  readonly evidenceItemIds: string[];  // Lineage: individual items
  readonly provenanceReferences: string[];  // Full provenance trail
  readonly sourceEvidenceIrIdentity: string;  // Root evidence collection ID
  readonly resolutionRuleId: string;  // Applied rule ID (e.g., "bgc-sem-rule-001")
  readonly resolutionRuleVersion: string;  // Rule version ("1.0.0")
  readonly certainty: { state: "certain" | "uncertain"; confidence: number };
  readonly conflictReferences: SemanticConflictReference[];  // Irreconcilable conflicts (preserved)
  readonly validationStatus: { valid: boolean; errors?: string[] };
  readonly resolutionContext: SemanticResolutionContext;  // Compiler metadata
  readonly diagnostics: CompilerDiagnostic[];  // Per-candidate diagnostics
}
```

**Supporting Types:**
- `SemanticEvidenceSignal` — Single evidence signal (5 fields)
- `SemanticAssertion` — Evidence-backed claim (6 fields)
- `SemanticConflictReference` — Irreconcilable conflict marker (4 fields)
- `SemanticResolutionRule` — Versioned classification rule (5 fields)
- `SemanticResolutionContext` — Compiler metadata (6 fields)
- `SemanticResolutionResult` — Per-cluster result (4 fields)
- `SemanticCandidateCollection` — M1.4 output (11 fields)

## 5. Resolution Rules Implemented

### 5.1 Rule Catalog (3 Signal Types)

| Signal Type | Extraction | Resolution Logic | Example |
|-------------|-----------|------------------|---------|
| **Category Match** | `metadata.category` | If category ∈ SEMANTIC_CLASSES → candidate | `{category: "constraint"}` → `SemanticCandidate(semanticClass="constraint")` |
| **Type Match** | `metadata.type` | If type ∈ SEMANTIC_CLASSES → candidate | `{type: "capability"}` → `SemanticCandidate(semanticClass="capability")` |
| **Semantic Class Match** | `metadata.semanticClass` | If semanticClass ∈ SEMANTIC_CLASSES → candidate | `{semanticClass: "customer"}` → candidate |

**Signal Priority:** semanticClass > category > type (applied in `extractSignalsFromMetadata()`)

### 5.2 Resolution Cases (Deterministic Router)

**Case 1: Unsupported BGS-0001 Class**
```
IF class ∈ BGS_VALID_BUT_UNSUPPORTED (goal, risk, decision, location, time, business-rule)
THEN no candidate + BGC-SEM-001 diagnostic
RATIONALE: Requires interpretation, not available in M1.4
```

**Case 2: Insufficient Explicit Evidence**
```
IF no signals extracted from metadata
THEN no candidate + BGC-SEM-002 diagnostic
RATIONALE: Only explicit signals count; no inference in M1.4
```

**Case 3: Multiple Conflicting Classes**
```
IF uniqueClasses.length ≥ 2
THEN ONE conflicted candidate with state="uncertain" + BGC-SEM-003 diagnostic
RATIONALE: Preserve conflict visibility; do not discard evidence
```

**Case 4: Single Explicit Class**
```
IF uniqueClasses.length == 1
THEN ONE candidate with state="certain" + resolutionRuleId + assertions
RATIONALE: Evidence-backed confidence; rule traceability
```

candidateId = deterministicIdentity(

## 16. Known Engineering Gaps

### 16.1 M1.5: Semantic Consolidation (Not in Scope)

**What M1.5 Will Do:**
- Consume `SemanticCandidateCollection` (M1.4 output)
- Merge same-class candidates from different clusters
- Emit `ConsolidatedSemanticEntityCollection`

**Why M1.4 Doesn't Do It:**
- Consolidation violates cluster independence (Invariant VII.1)
- Requires cross-cluster reasoning (belongs in subsequent pass)
- M1.4 is pure classification, not entity merging

### 16.2 M1.7: Relationship Resolution (Not in Scope)

**What M1.7 Will Do:**
- Resolve inter-entity relationships (customer → order → supplier)
- Build semantic relationship graph
- Emit `BusinessGenomeArtifact`

**Why M1.4 Doesn't Do It:**
- Relationships require consolidated entities (from M1.5)
- M1.4 has no context for relationship interpretation
- Graph building belongs after consolidation

### 16.3 Unsupported Classes (Supported in M1.5+)

**Why M1.4 Rejects goal, risk, decision, location, time, business-rule:**
- These require **interpretation** of evidence, not classification
- Example: "goal" → must infer whether evidence describes an objective
- M1.5 consolidation context provides inter-entity relationships that enable interpretation

## 17. Known Architecture Questions

### 17.1 BGS/BGC Relationship Taxonomy Discrepancy

**Observation:** BGS-0001 defines 19 semantic classes; M1.4 supports 13

**Discrepancy:** Some BGS classes (goal, risk, decision, location, time, business-rule) require interpretation, not explicit signals

**Resolution:** Per frozen spec, this is acknowledged but not addressed in M1.4. Design follows principle: **"Only explicit signals count; no inference."**

**Future Work:** M1.5+ may define inference rules (e.g., "if candidate.assertions contains stakeholder + expected outcome → classify as goal")
