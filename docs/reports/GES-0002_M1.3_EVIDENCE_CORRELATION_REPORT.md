GES-0002 / M1.3 implements the deterministic **Evidence Correlation** stage of the Business Genome Compiler. ## SECTION 4: CORRELATION MODEL

### Overview

The correlation model operates on the `GroupedEvidenceCollection` output from Evidence Grouping. Each `GroupedEvidenceSet` represents evidence items sharing identical grouping key material. Evidence Correlation determines which groups describe the same underlying enterprise reality and merges them into `EvidenceCluster` objects.

### Correlation Rules

Two correlation rules are implemented. Both are deterministic and use only information explicitly present in the validated compiler input.

#### Rule 1: Explicit IR Cross-Group Relationship

**Rule ID**: `bgc.correlation.rule.explicit-ir-cross-group-relationship`

If an explicit relationship exists in the Evidence IR between nodes belonging to different groups, those groups are correlated into the same cluster.

```
Evidence IR Relationship
    from: node in Group A
    to:   node in Group B
─────────────────────────
Group A and Group B → same EvidenceCluster
```

This rule is applied to all relationship types (`produced`, `derived_from`, `transformed_to`). The rule records:

- Relationship ID
- Relationship type
- `from` node ID
- `to` node ID

No inference is performed. The relationship must explicitly exist in the IR.

#### Rule 2: Identical Declared Subject Reference

**Rule ID**: `bgc.correlation.rule.identical-declared-subject-reference`

If two groups declare an identical non-empty `subjectReference` in their group key material, they are correlated. The `subjectReference` is a governed field in Evidence IR metadata that explicitly declares the subject of the evidence item.

```
Group A: deterministicGroupKey contains "subjectReference":"acme-corp"
Group B: deterministicGroupKey contains "subjectReference":"acme-corp"
──────────────────────────────────────────────────────────────────────
Group A and Group B → same EvidenceCluster
```

**Implementation note**: The `deterministicGroupKey` is produced by `stableStringify()`, which emits literal `undefined` (not `null`) for absent values, making the string invalid JSON. A targeted regex extraction is used instead of `JSON.parse` to reliably extract the `subjectReference` value:

```typescript
const match = /"subjectReference":"([^"]+)"/.exec(deterministicGroupKey);
```

#### No Additional Rules

No other correlation mechanisms are implemented. If correlation cannot be established by either rule, the group is preserved as a singleton cluster. No inference, no guess, no semantic interpretation.

### Cluster Formation Algorithm

A `DeterministicUnionFind` structure manages cluster membership:

```
Initialize all group IDs in union-find
Apply Rule 1 → union correlated groups
Apply Rule 2 → union correlated groups
Build clusters from union-find roots
Deterministic root selection: lexicographically smaller root wins
```

Root selection is deterministic: when two roots are merged, the lexicographically smaller root ID becomes the canonical root. This guarantees identical cluster membership regardless of union call order or processing sequence.

### Singleton Clusters

Groups that match no correlation rule each form their own singleton `EvidenceCluster`:

```
memberGroupIds:  [groupId]
correlationBases: []       (no correlation basis — singleton)
```

### Pass-Through Guarantee

Every Evidence Group from the input `GroupedEvidenceCollection` appears in exactly one `EvidenceCluster`. The pass validates this invariant and emits `BGC-CORR-004` if it is violated.

## SECTION 5: DETERMINISM GUARANTEES

The following determinism properties are verified by the test suite:

### 1. Stable Cluster IDs

Cluster IDs are computed via `deterministicIdentity("bgc-cluster", { deterministicClusterKey, root })`. The `deterministicClusterKey` is a `stableStringify` of `{ memberGroupIds, correlationRuleVersion, sourceEvidenceIrIdentity }`. Cluster IDs do not depend on timestamps, UUIDs, memory location, thread scheduling, or randomness.

**Test**: `"cluster IDs are stable and do not depend on timestamps or UUIDs"` — 5 runs produce identical IDs.

### 2. Stable Collection ID

The collection-level ID is computed as `deterministicIdentity("bgc-clusters", { sourceEvidenceIrIdentity, correlationRuleVersion, clusters })`. The full cluster array is hashed after sorting.

**Test**: `"collection ID is deterministic for equivalent inputs"`

### 3. Stable Output Ordering

Clusters are sorted by cluster ID (lexicographic). Within each cluster, `evidenceItemIds`, `memberGroupIds`, and `provenanceReferences` are sorted. Diagnostics are sorted by the existing `sortDiagnostics()` function.

**Test**: `"clusters are returned in deterministic lexicographic order"`  
**Test**: `"evidence item IDs within clusters are sorted deterministically"`  
**Test**: `"member group IDs within clusters are sorted deterministically"`

### 4. Permutation Independence

Reversing the order of groups in the input produces identical cluster output.

**Test**: `"permuted group input produces identical cluster output"`

### 5. Byte-Identical JSON

Identical inputs produce byte-identical `JSON.stringify` output.

**Test**: `"identical inputs produce byte-identical JSON output across runs"`

## SECTION 6: CONFLICT HANDLING

### Conflict Preservation

Conflicts are never silently resolved. The pass preserves conflict indicators throughout cluster formation and accumulates them from all member groups.

When correlated groups carry `explicitConflictIndicators`:

1. All conflict indicator strings are preserved on the resulting cluster
2. A `BGC-CORR-002` warning diagnostic is emitted at the cluster level
3. The conflicted cluster is passed unchanged to downstream stages
4. **Resolution is deferred to future governed compiler stages**

### Example

```
Group A: explicitConflictIndicators: ["conflict:duplicate-subject"]
Group B: explicitConflictIndicators: []
──────────────────────────────────────────────────────────────────
Cluster: explicitConflictIndicators: ["conflict:duplicate-subject"]
         diagnostic: BGC-CORR-002 (warning, not error — preserved, not blocked)
```

**Test**: `"conflict indicators from input groups are preserved on clusters"`

## SECTION 7: DIAGNOSTICS

### New Diagnostic Namespace

```
BGC-CORR-001  No groups to correlate (warning)
BGC-CORR-002  Cross-cluster conflict detected (warning)
BGC-CORR-003  Duplicate cluster assignment (error)
BGC-CORR-004  Evidence preservation mismatch (error)
```

### Diagnostic Descriptions

| Code | Severity | Condition |
|------|----------|-----------|
| `BGC-CORR-001` | warning | No evidence groups available for correlation (empty collection) |
| `BGC-CORR-002` | warning | A correlated cluster contains explicit conflict indicators from member groups |
| `BGC-CORR-003` | error | An evidence group was assigned to more than one cluster (invariant violation) |
| `BGC-CORR-004` | error | Number of unique evidence items in output clusters does not equal input group count |

### Diagnostic Structure

All diagnostics include:

- `code` — stable, namespaced code from `BGC_DIAGNOSTIC_CODES.CORR.*`
- `severity` — `"error"` or `"warning"`
- `passId` — `"bgc.evidence-correlation"` (always)
- `artifactId` — affected cluster or group ID (where applicable)
- `details` — structured detail object with affected identities

### Diagnostic Ordering

Diagnostics are merged from upstream passes and sorted deterministically by the existing `sortDiagnostics()` function (sorted by code, then passId, then artifactId, then message). No non-deterministic ordering is possible.

## SECTION 10: ENGINEERING GAPS

The following gaps are documented for architectural awareness. None are blocking for M1.3 completion.

### Gap 1: `_context` Parameter Linting

**Description**: All genome compiler passes declare `_context: CompilerPassContext` but do not use it. ESLint warns on each. The underscore prefix convention suppresses errors in most configurations but not the current ESLint setup.

**Severity**: Minor  
**Resolution**: Consistent with baseline; ESLint configuration change outside this milestone scope.

### Gap 2: `stableStringify` Non-JSON Output

**Description**: `stableStringify()` emits literal `undefined` for absent values, producing strings that are not valid JSON. This required the regex-based `extractSubjectReference` rather than `JSON.parse`.

**Severity**: Low  
**Impact**: Contained to `extractSubjectReference` function; handled correctly.  
**Resolution**: If stableStringify is updated to emit `null` for absent values, the regex approach remains correct and is forward-compatible.

### Gap 3: Rule Coverage Limited to Explicit Declarations

**Description**: By design, only two correlation rules are implemented. Implicit correlations (e.g., semantic similarity, entity matching) are out of scope for M1.3 and appropriately belong to Semantic Resolution (M1.4).

**Severity**: None (by design)  
**Resolution**: This is the correct architectural boundary.

### Gap 4: No Cross-File Integration Test

**Description**: There are no integration tests running a complete evidence PDF through the pipeline from Discovery → Evidence IR → Business Genome Compiler including M1.3. All tests use fixture Evidence IR data.

**Severity**: Low  
**Resolution**: End-to-end integration testing can be added alongside M1.4 as the pipeline matures.
